## 结论

**Goal-driven 是合适的上层业务抽象，但“数字员工可以自由、递归地把 Goal 委托给其他员工”不是企业平台的最优默认实现。**

更适合你这个平台的方案是：

> **Goal 驱动 + 负责人持续持有责任 + Platform 受控编排 + 动态执行图 + 有边界的专业员工协作**

也可以概括为：

> **Agent 提议怎么协作，Platform 决定是否允许，并负责可靠执行。**

文档当前的流程是“负责人拆解目标、调用专业员工、收集 Artifact、最终汇总”，本质上更接近 **Orchestrator–Worker（负责人—执行者）**，而不是责任不断向下转移的 Handoff。这个方向是正确的，但风险控制目前明显不足。

---

## 一、当前“风险缓解”只解决了可观察性，没有解决可控性

现在第 11 章写的是：

> Goal 委托链不可控 → 全量记录 Delegation Log 和 Artifact

这只能回答：

* 发生了什么；
* 谁委托了谁；
* 哪个员工提交了什么；
* 出错后怎么复盘。

但它阻止不了：

* A 委托 B，B 委托 C，C 又委托 A；
* 一个负责人一次生成几十个子目标；
* 多个员工重复执行同一任务；
* 子目标不断变细，始终不收敛；
* Token、时间和并发量持续膨胀；
* 敏感上下文跨部门传播；
* 委托员工离线后 Goal 永久卡住；
* 子员工再次委托后，根负责人失去实际控制；
* 写操作被拆成多个低风险子任务，从而绕开审批；
* 平台重启后不知道执行到哪一步。

Anthropic 在其多 Agent 实践中也观察到过类似问题，例如简单问题生成大量子 Agent、重复搜索、任务边界模糊等；其生产架构最终采用的是 Lead Agent 统一协调的 orchestrator-worker 模式，而且多 Agent 的 Token 消耗可能显著高于普通单 Agent 交互。([Anthropic][1])

所以这里不能只写“记录日志”，而应当把风险拆成：

1. **执行拓扑失控**
2. **成本与资源失控**
3. **权限和数据传播失控**
4. **责任归属失控**
5. **状态和故障恢复失控**
6. **结果质量逐级衰减**

---

## 二、不要把 Delegation 和 Handoff 混为一谈

建议在领域模型里明确区分三种模式。

### 1. Handoff：责任转交

A 把任务移交给 B，之后 B 成为当前负责人。

适合：

* 客服分流；
* 售后问题转退款专员；
* 用户下一阶段直接与专业员工沟通。

OpenAI Agents SDK 中的 Handoff 就是这种语义：专业 Agent 接管后续执行；而“Agent as Tool”则是负责人调用专业 Agent 完成一个有限子任务，负责人继续控制流程和最终结果。([OpenAI][2])

### 2. Delegation：有限子任务委托

A 仍然是 Goal Owner，只把一个明确的 Work Item 分配给 B：

* B 不接管根 Goal；
* B 只提交指定格式的 Artifact；
* A 负责验收、整合和最终完成；
* B 是否允许继续委托，由 Platform Policy 决定。

**你的平台应该默认采用这一种。**

### 3. Workflow：平台代码编排

对于稳定、强合规、步骤固定的业务，由 Platform 决定执行顺序，例如：

```text
读取订单
→ 验证退款条件
→ 生成退款建议
→ 人工审批
→ 执行退款
→ 通知客户
```

这种任务不应该让 Agent 自由生成委托链。

Anthropic 对 agentic system 的建议也是先采用最简单、可组合的模式：固定流程适合 Workflow，只有步骤无法预先确定时，才使用 Agent 自主规划。([Anthropic][3])

---

## 三、最推荐的模型：动态执行图，而不是自由委托链

文档中“不靠预设 DAG”这个判断可以保留，但需要增加一句：

> **不使用预设静态 DAG，不等于运行过程中没有执行图。**

Platform 应该维护一个运行时动态生成的 Goal Execution Graph：

```text
Root Goal
│
├── Work Item A：市场数据收集
│   ├── Artifact A1
│   └── Artifact A2
│
├── Work Item B：客户数据分析
│   └── Artifact B1
│
└── Work Item C：报告撰写
    ├── depends_on: A
    └── depends_on: B
```

这个图可以由负责人动态提出，但必须由 Platform 创建和维护。

这样既保留 Agent 的灵活性，又能实现：

* 防止环路；
* 控制深度；
* 控制扇出；
* 管理依赖关系；
* 并行执行；
* 取消整棵任务树；
* 失败重试；
* 超时处理；
* 成本统计；
* 图形化追踪。

因此，“Goal 委托链”这个词也可以改为：

> **Goal 动态执行图失控**

因为真实协作通常不是单链条，而是分叉、并行、汇合形成的图。

---

## 四、Platform 必须是所有委托的强制中介

不建议允许 Hermes Instance 直接调用另一个 Hermes Instance。

应该统一经过：

```text
Hermes Employee
    │
    │ propose_delegation
    ▼
Platform Goal & Delegation Gateway
    │
    ├── 身份校验
    ├── Goal 状态校验
    ├── 委托策略校验
    ├── 环路检测
    ├── 预算校验
    ├── 数据权限校验
    ├── 审批判断
    └── 创建 Work Item
           │
           ▼
    Target Hermes Instance
```

关键原则是：

> **Agent 只有提出委托的能力，没有直接调度其他员工的能力。**

平台可以给 Hermes 暴露一个 `delegate_work_item` Business Tool，但这个 Tool 只能调用 Platform API。

## 五、每次委托必须形成结构化合同

当前 `delegate_task()` 里的 `sub_goal: str` 和 `context: str` 太自由，容易出现目标模糊、重复执行和上下文泄露。

建议把委托建模成结构化 `WorkItem`：

```python
class WorkItem:
    id: str
    root_goal_id: str
    parent_work_item_id: str | None

    owner_employee_id: str
    delegator_employee_id: str
    assignee_employee_id: str

    objective: str
    acceptance_criteria: list[str]
    expected_artifact_schema: dict

    input_artifact_refs: list[str]
    context_refs: list[str]
    data_classification: str

    token_budget: int
    deadline: datetime
    max_retries: int

    depth: int
    can_redelegate: bool

    status: str
```

其中最重要的是四项：

### `objective`

明确要求完成什么，而不是一句模糊自然语言。

### `acceptance_criteria`

负责人如何判断子目标完成，例如：

```json
[
  "至少覆盖最近三个季度",
  "所有结论必须引用数据来源",
  "输出必须包含同比与环比",
  "不得直接修改 CRM 数据"
]
```

### `expected_artifact_schema`

让子员工提交结构化结果，而不是一段无法稳定消费的自由文本。

### `context_refs`

尽量传递 Artifact、Knowledge Citation、业务对象引用，不要直接把完整父级对话和全部上下文复制给子员工。

---

## 六、委托策略应当由 Platform 强制执行

建议新增 `DelegationPolicy`：

```python
class DelegationPolicy:
    max_depth: int
    max_children_per_node: int
    max_total_work_items: int
    max_parallelism: int
    max_retries: int

    allow_redelegation: bool
    allow_cross_department: bool

    goal_token_budget: int
    work_item_token_budget: int
    execution_timeout_seconds: int

    approval_rules: list[str]
```

### MVP 推荐默认值

| 控制项                 |              MVP 建议 |
| ------------------- | ------------------: |
| 默认委托深度              |                   1 |
| 最大委托深度              |                   2 |
| 单节点最大子任务数           |                 3～5 |
| 单 Goal 最大 Work Item |                  10 |
| 并发执行数               |                   3 |
| 默认重试次数              |                   1 |
| 子员工再次委托             |                默认关闭 |
| 跨部门敏感委托             |                需要审批 |
| 写操作                 | 继续由 Tool Gateway 审批 |

这些数值应当可以按 Goal 类型或岗位模板调整，而不是写死在代码中。

特别建议：

> **Phase 2 先只支持一层委托。**

也就是负责人可以委托专业员工，但被委托员工不能再次委托。等追踪、预算、权限和恢复机制稳定以后，再开放受控的二层委托。

---

## 七、根 Goal 的责任人不能因为委托而变化

必须有一个不可变规则：

> **Root Goal Owner 对最终结果持续负责。**

即使下面发生多次委托：

```text
经理 A
  → 分析师 B
      → 数据员工 C
```

也不能变成“现在 C 是整个 Goal 的负责人”。

正确语义应该是：

* A 是 Root Goal Owner；
* B 是 Work Item B 的 Assignee；
* C 是 Work Item C 的 Assignee；
* B 对 C 的结果进行局部验收；
* A 对整个 Goal 最终验收；
* Platform 始终知道责任链，但不会转移根责任。

否则 KPI 会产生严重歧义：

* 最终结果错误算谁的？
* B 委托错误还是 C 执行错误？
* A 是否履行了验收职责？
* 子员工失败是否直接扣根负责人 KPI？

建议 KPI 分为：

* `goal_ownership_score`
* `delegation_quality_score`
* `work_item_execution_score`
* `artifact_acceptance_rate`
* `rework_rate`
* `delegation_cost_efficiency`

不要简单地把子 Goal 失败全部归因给最后一个执行员工。

---

## 八、权限不能沿委托链传播

这是企业场景中比 Token 超支更严重的风险。

必须明确：

> **委托不会产生权限继承，也不会产生权限扩大。**

例如销售经理把任务委托给财务员工：

* 财务员工使用自己的 Employee Service Token；
* 财务员工只能调用自己岗位允许的 Tool；
* 不能使用销售经理的 Token；
* 不能因为 Goal 来自经理就自动获得经理权限；
* 经理也不能通过财务员工绕过自身权限读取原始财务数据。

建议分开处理三种权限：

### 执行权限

```text
实际权限 =
目标员工自身 Tool Entitlement
∩ Goal Policy
∩ 当前 Work Item 约束
```

### 上下文传递权限

```text
可传递上下文 =
委托方允许分享的数据
∩ 接收方允许接收的数据
∩ 跨部门数据策略
```

### Artifact 返回权限

子员工可以返回经过汇总或脱敏的结果，而不一定返回全部原始数据。

例如财务员工可以返回：

```json
{
  "credit_level": "medium",
  "recommended_limit": 50000
}
```

但不把客户全部银行流水传给销售员工。

## 九、Goal 必须有独立预算，不能只依赖员工配额

当前文档已有 Organization、Department、Digital Employee 三层配额，这是必要的，但不足以控制委托扩散。

还应增加：

* Root Goal Token Budget；
* 单 Work Item Token Budget；
* 最大模型调用次数；
* 最大 Tool 调用次数；
* 最大并发子任务数；
* Deadline；
* 可选成本预算。

例如：

```text
Root Goal Budget: 200,000 tokens

├── 市场调研：50,000
├── 数据分析：70,000
├── 报告撰写：50,000
└── 保留预算：30,000
```

这里不一定要违反文档中“不按预估 Token 预扣账”的原则：

* Token Ledger 仍以真实 usage 结算；
* Goal Budget 是执行上限，不是财务记账；
* 每次模型调用完成后更新 Goal 已用量；
* 下一次调用前判断是否已经达到上限；
* 并行子任务可以做软预算预留，防止多个分支同时认为自己可以消费全部剩余预算。

多 Agent 更适合高价值、可并行、开放性任务；对于价值较低或依赖关系非常强的任务，额外 Token 和协调成本可能不划算。([Anthropic][1])

## 十、需要比当前状态机更完整的状态

当前状态：

```text
created → in_progress → blocked → completed / failed
```

对于异步委托过于简单。

### Goal Run 建议状态

```text
created
→ planning
→ running
→ waiting_approval
→ blocked
→ completed
→ failed
→ cancelled
→ expired
```

### Work Item 建议状态

```text
proposed
→ authorized
→ assigned
→ running
→ submitted
→ accepted
→ completed
```

异常分支：

```text
rejected
rework_required
blocked
failed
cancelled
expired
```

其中 `submitted` 和 `accepted` 必须分开：

* 子员工提交 Artifact 不代表任务已经完成；
* 必须经过负责人或验证器验收；
* 验收失败可以进入 `rework_required`；
* 达到重试次数后再进入 `failed`。

## 十一、可靠执行不能依赖 Hermes 会话本身

长时间 Goal 可能持续数分钟、数小时，甚至等待人工审批。Platform 需要持久化：

* Goal Run；
* Work Item；
* 状态变更事件；
* Artifact；
* Approval；
* Retry；
* Timeout；
* Policy Decision；
* Hermes run/session ID。

每次状态变化都应该先写入 Platform，再触发下一步执行。

可以先使用：

```text
PostgreSQL + Outbox Event + Worker + Redis Queue
```

MVP 不一定立即引入 Temporal，但数据模型要具备“可恢复执行”的语义。Temporal 的 Durable Execution 和 Event History 就是这种思路：执行进度持久化后，即使进程、网络或 Worker 出现故障，也能够恢复，而不是依赖一个常驻内存中的调用链。([Temporal 文档][4])

## 十二、追踪仍然重要，但应该是最后一道能力

Delegation Log 应升级为完整 Trace：

```text
goal_run_id
trace_id
work_item_id
parent_work_item_id
delegator_employee_id
assignee_employee_id
hermes_run_id
tool_call_id
approval_id
artifact_id
policy_decision_id
```

管理后台应该展示：

* 动态任务图；
* 当前活动节点；
* 等待审批节点；
* 失败节点；
* 每个节点 Token 消耗；
* 委托理由；
* 输入和输出 Artifact；
* 重试次数；
* 谁取消了任务；
* 权限或预算拒绝原因。

成熟 Agent 框架也会将 Agent 调用、模型生成、Tool 调用、Handoff 和 Guardrail 记录在同一个 Trace/Span 体系中，以便开发和生产排障。([OpenAI][5])

但要强调：

> Trace 是“看见失控”，Policy Gateway 才是“阻止失控”。

## 十三、什么任务适合 Goal-driven Delegation

### 非常适合

* 企业调研；
* 跨部门信息汇总；
* 竞品分析；
* 多数据源报告；
* 方案设计；
* 需要多个专业角色独立分析的复杂问题；
* 可以明显并行的任务。

### 不适合自由委托

* 固定审批流程；
* 工资计算；
* 财务过账；
* 退款执行；
* 合同正式签署；
* 权限变更；
* 数据删除；
* 合规报送；
* 强顺序、强一致性的业务流程。

这些任务更适合：

```text
固定 Workflow
+ Agent 完成局部认知任务
+ Platform 决定流程
+ 人工审批关键动作
```

敏感 Tool 调用还应支持暂停、审批、恢复，而不是允许子员工自动继续。类似的 HITL 模式会将待审批调用持久化，在审批完成后恢复原执行。([OpenAI][6])

## 十四、我建议修改文档中的架构原则

目前是：

> 多 Agent 协作不靠预设 DAG，而靠 Goal-driven Delegation。

建议改成：

> 多 Agent 协作不依赖预设静态 DAG。负责人可以根据 Goal 动态提出子目标，Platform 将通过 Delegation Policy 校验后创建受控 Work Item，并维护可追踪、可取消、可恢复的动态执行图。Root Goal Owner 始终对最终结果负责；子员工默认不得继续委托。

同时将第 11 章风险项改为：

| 风险           | 影响                                           | 缓解                                                                                                                                                            |
| ------------ | -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Goal 动态执行图失控 | 递归委托、环路、任务爆炸、重复执行、预算超限、上下文越权、责任不清和 Goal 无法收敛 | 所有委托统一经过 Platform Gateway；Root Owner 不变；采用结构化 Work Item 和动态执行图；限制深度、扇出、总节点数、并发、重试、预算和 Deadline；默认禁止子员工再次委托；执行环路检测、权限校验、敏感委托审批、Artifact 验收、全链路 Trace、超时取消和故障恢复 |

---

## 最终判断

对于你的企业数字员工平台：

* **Goal-driven 应该保留**，它非常适合作为人类向数字员工分配工作的产品入口。
* **负责人自主拆解也应该保留**，这是 Agent 相对于固定流程的重要价值。
* **自由递归委托不应成为默认能力**。
* **MVP 最好采用单层 Orchestrator–Worker**。
* **后续升级为受 Platform 管控的动态执行图**。
* **确定性业务继续使用 Workflow，不要强行 Agent 化**。
* **Handoff 只用于明确的责任转交场景，不用于一般子任务协作**。

最核心的一句话是：

> **Goal 是业务意图，Delegation 是受控执行机制；不能让自然语言 Goal 同时承担权限、预算、流程和可靠性控制。**

## 我的总体判断

**我会采购，但不会把它当成“自动替代员工的软件”采购，而会把它当成“可治理的数字化工作执行平台”采购。**

我愿意买的前提是：

> 它能够把企业中大量“需要理解、判断、查资料、组织信息，但最终结果可以复核”的工作，变成可分配、可追踪、可验收、可审计的数字化工作。

我不会因为它能创建几十个 Agent、能自动委托任务而采购。相反，Agent 越多、委托链越复杂，我越谨慎。

企业真正愿意付钱购买的，不是“AI 自主性”，而是：

* 可量化的业务产出；
* 更短的业务周期；
* 更低的协作成本；
* 更稳定的服务质量；
* 对权限、责任和风险的控制。

目前企业 AI 的现实也说明了这一点：2025 年 McKinsey 调查中，88% 的受访组织已经在至少一个职能中经常使用 AI，62% 至少在试验 AI Agent，但接近三分之二尚未进入企业级规模化阶段，只有39%报告了企业层面的 EBIT 影响。真正获得较高价值的企业，通常不是简单部署工具，而是重新设计工作流程、建立人工验证机制并持续追踪 KPI。([McKinsey & Company][1])

# 一、什么样的任务适合 AI 数字员工

我会用六个问题判断一个任务是否适合：

1. **输入是不是数字化的？**
2. **主要工作是不是阅读、分析、生成、查询和协调？**
3. **任务是否经常发生，而不是一年一次？**
4. **结果是否可以被人或规则验证？**
5. **出错后是否可以撤销或修正？**
6. **是否能明确计算时间、成本或质量收益？**

满足得越多，越适合数字员工。

最理想的任务通常具有以下特征：

> **高频、耗时、知识密集、输入不完全固定、输出可验证、行动可逆。**

不理想的任务则是：

> **低频、不可逆、高法律责任、高人身风险、目标模糊且无法验证。**

---

# 二、我最愿意让数字员工承担的七类工作

## 1. 企业情报与管理分析

这是我最愿意首先采购的场景。

例如：

* 每日行业新闻监测；
* 竞争对手产品、价格和市场动作分析；
* 政策法规变化跟踪；
* 客户舆情和投诉趋势分析；
* 周报、月报、经营分析材料准备；
* 多部门数据和材料汇总；
* 管理会议前的 Briefing；
* 针对异常指标查询相关原因；
* 给管理层形成决策选项，而不是直接替管理层决策。

一个经营分析数字员工可以把原来需要多人花一两天准备的材料，先形成：

```text
本周经营异常
├── 销售额下降区域
├── 可能原因
├── 客户反馈证据
├── 竞争对手近期动作
├── 需要管理层确认的问题
└── 建议采取的三个行动
```

它的价值不是“替 CEO 做决定”，而是：

> **让管理者把时间从搜集和整理信息，转移到判断、取舍和承担责任。**

知识管理和深度研究也是当前企业 AI Agent 应用较多的职能之一。([McKinsey & Company][1])

---

## 2. 客户服务和售后协同

适合的任务包括：

* 客户问题分类；
* 查询订单、产品和服务记录；
* 从知识库查找答案；
* 生成回复草稿；
* 判断应转给哪个部门；
* 归纳历史沟通记录；
* 识别高风险投诉；
* 创建工单草稿；
* 跟踪工单超时；
* 总结投诉根因。

数字员工可以形成协作：

```text
客服协调员工
├── 查询订单员工
├── 产品知识员工
├── 售后政策员工
└── 投诉风险员工
```

但初期不应让它直接执行：

* 无审批退款；
* 修改合同；
* 承诺赔偿；
* 删除客户信息；
* 对重大投诉作出最终认定。

更合理的模式是：

```text
AI 查询和分析
→ AI 生成建议
→ 人工确认
→ 系统执行
```

这类场景可以显著降低客服查询、整理和转派所需时间，同时保持人类对高风险决定的控制。

---

## 3. 销售支持，而不是完全自动销售

适合数字员工完成：

* 潜在客户背景调查；
* 客户所在行业分析；
* 汇总客户历史交流；
* 识别潜在需求；
* 推荐销售话术；
* 准备拜访材料；
* 生成方案书初稿；
* 生成报价说明；
* 标书信息整理；
* CRM 数据质量检查；
* 商机跟进提醒；
* 会后纪要和待办事项。

例如，销售人员第二天拜访一家制造企业，数字员工可以提前准备：

```text
客户基本情况
行业近期变化
客户可能面临的问题
历史采购记录
关键联系人
竞争对手关系
建议提问清单
可推荐产品
风险和禁区
```

它创造的主要价值不是替销售“说服客户”，而是：

> **让普通销售人员拥有接近优秀销售助理和行业研究员的准备能力。**

但不建议数字员工自行：

* 对外承诺价格；
* 修改信用额度；
* 签署合同；
* 提供未经批准的折扣；
* 自动群发高风险营销内容；
* 对客户作出法律或财务承诺。

---

## 4. 运营和项目管理

这是企业内部很有价值、但经常被忽略的场景。

数字员工可以承担：

* 会议材料准备；
* 会议纪要；
* 决策事项提取；
* 责任人和截止日期整理；
* 跨部门事项催办；
* 项目风险汇总；
* 里程碑偏差识别；
* 重复问题归类；
* 项目周报；
* SOP 执行检查；
* 异常事项升级；
* 不同项目材料格式统一。

很多企业的问题不是员工不会工作，而是：

* 会议决定没有持续跟踪；
* 信息散落在邮件、聊天、文档和业务系统中；
* 部门之间不断询问进度；
* 管理者不知道项目真实状态；
* 大量时间浪费在催办和汇总上。

数字员工在这里创造的是：

> **降低组织内部的协调税。**

这可能比单纯节省文案编写时间更有商业价值。

---

## 5. IT 服务和研发辅助

适合的工作包括：

* IT 服务台问题分类；
* 根据知识库提供解决步骤；
* 日志初步分析；
* 故障信息汇总；
* 相似故障检索；
* 创建故障工单；
* 生成测试用例；
* 代码解释；
* 代码审查建议；
* API 文档整理；
* 版本变更说明；
* 技术债务清单；
* 开发任务拆解；
* 故障复盘材料准备。

但数字员工不应默认拥有：

* 生产数据库写权限；
* 生产环境任意 Shell 权限；
* 无审批发布权限；
* 删除资源权限；
* 修改安全策略权限；
* 直接处理密钥的权限。

对于运维场景，我更愿意采用：

```text
数字员工诊断
→ 生成建议命令
→ 人工确认
→ 受控工具执行
→ 记录结果
```

而不是允许 Agent 自行在生产环境不断尝试。

## 6. 财务、法务和人力资源的“辅助层”

这些部门可以使用数字员工，但必须限制在辅助层。

### 财务

适合：

* 发票信息提取；
* 报销材料完整性检查；
* 对账差异识别；
* 异常交易提示；
* 预算执行情况汇总；
* 财务报告初稿；
* 付款申请材料整理。

不适合自主完成：

* 最终付款；
* 财务记账确认；
* 税务申报；
* 信贷批准；
* 重大资产处置。

### 法务

适合：

* 合同条款提取；
* 标准条款对比；
* 缺失条款提醒；
* 历史合同检索；
* 法规变化摘要；
* 风险问题清单；
* 法务审查材料准备。

不适合：

* 代表企业给出最终法律意见；
* 自主签署合同；
* 决定是否诉讼；
* 对外作出法律承诺。

### 人力资源

适合：

* 入职材料检查；
* 制度问答；
* 培训内容推荐；
* 简历信息结构化；
* 面试记录总结；
* 员工服务工单分派；
* 岗位说明书起草。

不适合完全自主决定：

* 是否录用；
* 是否晋升；
* 是否辞退；
* 薪酬调整；
* 纪律处分。

因为这些决定涉及公平、歧视、法律责任以及个人职业命运，数字员工只能提供辅助材料，不能成为责任主体。

## 7. 企业知识管理

这是你当前平台最有可能快速落地的场景之一。

你已经设计了：

* Knowledge Source；
* Knowledge Entitlement；
* RAGFlow Adapter；
* 引用来源；
* 动态权限校验；
* 检索审计；
* 知识源失联处理。

这些设计使数字员工不只是一个公共大模型聊天入口，而有可能成为：

> **理解岗位、知道企业内部制度、只能访问授权知识范围的专业员工。**

它可以承担：

* 新员工制度问答；
* 产品知识问答；
* SOP 查询；
* 历史项目经验查询；
* 故障案例检索；
* 标准合同和模板定位；
* 部门知识总结；
* 知识冲突和过期内容识别；
* 将工作过程中产生的新经验整理为待审核知识。

这类任务的好处是：

* 风险相对可控；
* 输出容易验证；
* 使用频率高；
* 能覆盖多个部门；
* 容易测量搜索时间和解决率。

# 三、数字员工真正能创造的价值

## 1. 释放人的时间

最直接的价值是减少：

* 查资料；
* 整理资料；
* 复制数据；
* 格式转换；
* 写初稿；
* 汇总进度；
* 反复询问；
* 低价值沟通。

但不能只计算“节省了多少小时”。

因为节省出来的时间不一定自动变成利润。2026 年一项针对法国中型企业负责人的调查显示，77%的企业已经使用生成式 AI，但只有17%的使用者报告了时间节省，说明“安装了 AI”与“获得可测量生产力”之间存在很大距离。([Reuters][2])

采购时要继续问：

> 节省出来的时间，最终转化成了更多客户、更快交付、更少错误，还是只是让员工处理了更多无关工作？

---

## 2. 缩短业务周期

很多 AI 项目的真正价值不是减少人员，而是缩短周期：

* 客户回复从数小时缩短到数分钟；
* 投标材料从五天缩短到两天；
* 故障初步定位从两小时缩短到二十分钟；
* 管理报告从每月一次变成每日更新；
* 新员工熟悉制度从数周缩短到数天；
* 合同初筛从一天缩短到十分钟。

周期缩短可能带来：

* 更高成交概率；
* 更快回款；
* 更低客户流失；
* 更少项目延期；
* 更及时的风险处理。

---

## 3. 降低组织协调成本

企业越大，协调成本越高。

数字员工可以持续处理：

* 谁负责；
* 进度如何；
* 缺什么材料；
* 谁还没有反馈；
* 哪项任务已经超时；
* 哪个部门存在依赖；
* 下一步应该通知谁。

这种价值不一定能立刻表现为裁员，但能明显提升组织运行速度。

---

## 4. 把优秀员工的方法复制给更多人

优秀员工通常拥有：

* 更好的问题清单；
* 更好的分析框架；
* 更好的信息来源；
* 更好的工作顺序；
* 更准确的风险判断；
* 更成熟的文档模板。

通过岗位模板、Skill、知识源和工具权限，可以把这些方法部分固化到数字员工中。

于是，新员工或经验较少的员工也可以获得类似的工作支持。

这不是复制优秀员工的“判断力”，而是复制其：

> **工作准备、信息获取、标准动作和检查清单。**

---

## 5. 提高一致性和可追溯性

人类员工有时会：

* 忘记执行某个步骤；
* 没有引用资料来源；
* 使用过期模板；
* 漏掉审批；
* 没有留下完整记录；
* 因经验不同导致交付差异。

数字员工平台如果有统一的：

* Job Template；
* Skill；
* Knowledge Entitlement；
* Tool Entitlement；
* Approval；
* Artifact；
* Audit；
* Quota；

就可以提高流程一致性，并让每次工作有迹可循。你的方案已经具备这些核心治理模块，这是它比“给每个人开一个聊天机器人账号”更有采购价值的地方。

---

## 6. 提供弹性产能

企业经常出现临时工作高峰：

* 月末和季末；
* 大型投标；
* 产品发布；
* 客户投诉集中爆发；
* 政策突然变化；
* 系统故障；
* 集中审计。

数字员工可以提供一定的弹性知识工作产能，而不必为了短期高峰立即扩大固定人员规模。

但是这种弹性不是无限的，它仍受：

* 模型成本；
* 并发能力；
* 人工审核能力；
* 知识质量；
* 下游业务系统容量；

等因素限制。

---

# 四、我最担心的风险

## 1. “看起来完成了”，实际上结果错误

这是最常见的风险。

数字员工可能：

* 引用不存在的事实；
* 使用错误数据；
* 忽略关键限制；
* 混淆不同客户；
* 对模糊任务过度推断；
* 用流畅语言掩盖证据不足。

McKinsey 2025 年调查中，51%的 AI 使用组织报告至少出现过一种负面后果，接近三分之一的受访者报告过由 AI 不准确造成的后果。([McKinsey & Company][1])

所以平台不能只记录“任务已完成”，而要记录：

* 依据了哪些数据；
* 引用了哪些知识；
* 使用了什么工具；
* 哪些结论是推断；
* 谁进行了验收；
* 验收标准是什么。

---

## 2. 人工审核反而成为新瓶颈

AI 可以快速生成大量内容，但人类未必有足够时间审核。

结果可能是：

```text
过去：人写 10 份材料
现在：AI 写 100 份材料，人审核 100 份
```

表面产出提高了，实际组织负担可能更大。

因此不能只看 AI 的产出数量，要看：

* 一次验收通过率；
* 人工审核分钟数；
* 返工率；
* 错误逃逸率；
* 无实际用途的产物比例。

---

## 3. 权限和敏感数据泄露

多数字员工协作后，风险不只是“某个 Agent 越权”，还包括：

* 上下文被委托给无权员工；
* 敏感信息出现在 Artifact 中；
* 不同部门的数据被错误合并；
* 日志记录了明文隐私数据；
* 外部模型收到内部资料；
* Agent 借助其他 Agent 间接绕过权限。

你的平台必须坚持：

> 委托任务不等于委托权限。

接收任务的数字员工只能使用自身权限，不能继承委托者权限。

---

## 4. 自动执行造成真实业务损失

当数字员工从“给建议”升级为“直接操作系统”，风险会明显增加。

例如：

* 错发邮件；
* 错改客户状态；
* 重复创建订单；
* 错误退款；
* 删除数据；
* 错误发布代码；
* 误停服务；
* 使用错误价格报价。

企业 AI 风险并非理论问题。EY 对大型企业的调查显示，许多 AI 部署已经产生由合规失败、错误输出、偏差等导致的初始财务损失，而治理机制更成熟的企业在销售、成本节约和员工满意度方面表现更好。([Reuters][3])

因此我会要求：

* 默认只读；
* 写操作明确标识；
* 高风险操作必须审批；
* 参数有范围约束；
* 操作可幂等；
* 关键操作可撤销；
* 全程留痕；
* 可以快速停止。

---

## 5. 责任被推给 AI

失败后最危险的一句话是：

> “是 AI 做的。”

企业不能接受这种责任逻辑。

必须始终明确：

* Root Goal 谁负责；
* 谁批准了高风险操作；
* 谁验收了 Artifact；
* 哪个岗位模板授权了工具；
* 哪条规则允许了执行；
* 哪位管理员发布了配置。

AI 可以是执行主体，但不能成为法律和管理意义上的最终责任主体。

---

## 6. 大量低价值 Agent 导致组织复杂化

企业可能很快创建：

* 市场数字员工；
* 销售数字员工；
* 客服数字员工；
* 分析数字员工；
* 报告数字员工；
* 审核数字员工；
* 调度数字员工。

最后没有人知道：

* 它们分别负责什么；
* 哪些能力重复；
* 哪些仍有人使用；
* 哪些结果可信；
* 哪些配置已经过期；
* 成本为什么越来越高。

所以数字员工也需要完整生命周期：

```text
申请
→ 评审
→ 试运行
→ 发布
→ 监控
→ 定期复评
→ 停用
→ 归档
```

不能让各部门无限创建。

---

## 7. 员工能力退化和不当依赖

如果员工长期把分析、写作和判断全部交给 AI，可能逐渐失去：

* 独立思考能力；
* 业务基本功；
* 对异常的敏感度；
* 对结果质量的判断能力。

Microsoft 2026 年调查中，较成熟的 AI 使用者普遍强调把 AI 输出视为起点而非最终答案，并保持人类对思考和结果的责任；质量控制和批判性思维也被列为更重要的人类能力。([Microsoft][4])

因此企业不应要求员工“完全相信数字员工”，而应训练员工成为：

> 任务定义者、结果评审者、异常处理者和责任承担者。

# 五、我会不会采购你设计的这套平台

## 我的答案是：会，但先采购试点，不会立即全公司铺开

你现在的方案比普通 Agent 平台更吸引我的地方是，它不是只强调 Agent 聊天和自主执行，而是已经考虑：

* 岗位模板；
* 一个员工一个隔离运行时；
* 知识权限；
* 工具白名单；
* 参数约束；
* Credential 集中管理；
* 审批；
* 审计；
* Token 配额；
* Artifact；
* KPI；
* Goal 与 Delegation。

这些都是企业采购关心的内容。

但是，在我签正式企业合同之前，我还会要求平台补齐几个关键能力。

---

## 六、我采购前的八项硬性条件

### 1. 必须从具体业务场景开始

不能只演示：

> “创建一个营销员工，它会自动找其他员工完成任务。”

我要看到：

```text
具体业务对象
具体输入
具体输出
明确验收标准
明确人工责任人
明确成本
明确基线
```

---

### 2. 必须有任务风险分级

建议把任务分成四级：

| 级别       | 数字员工权限    | 示例         |
| -------- | --------- | ---------- |
| L1 信息辅助  | 查询、总结、起草  | 知识问答、报告初稿  |
| L2 工作协同  | 创建内部任务和草稿 | 工单草稿、项目催办  |
| L3 受控执行  | 审批后写业务系统  | 发邮件、修改 CRM |
| L4 高风险决策 | 原则上由人决定   | 付款、辞退、签约   |

不同级别不能使用相同的执行策略。

---

### 3. 必须支持人类验收，而不只是审批

审批关注的是：

> 允许不允许执行。

验收关注的是：

> 做得对不对，是否达到要求。

平台需要支持：

```text
Artifact submitted
→ Human/Rule validation
→ Accepted
或
→ Rework required
```

---

### 4. 必须具备端到端业务指标

不能只展示：

* 调用了多少次模型；
* 消耗了多少 Token；
* 完成了多少 Goal；
* 创建了多少数字员工。

我要看到：

* 业务周期减少多少；
* 人工处理时间减少多少；
* 首次验收通过率；
* 每单成本；
* 返工率；
* 错误率；
* 客户满意度；
* 增量收入；
* 风险事件数量。

---

### 5. 必须可以快速停机和降级

发生问题时应支持：

* 停止某个 Goal；
* 停止某个员工；
* 禁用某个 Tool；
* 撤销 Employee Service Token；
* 关闭自动委托；
* 切换为只读模式；
* 切换为只生成草稿；
* 回滚岗位模板和 Skill；
* 保留现场供审计。

---

### 6. 必须限制委托复杂度

我会要求 MVP：

* 只允许一层委托；
* 子员工默认不能继续委托；
* 每个 Goal 有总预算；
* 有最大节点数；
* 有并发限制；
* 有 Deadline；
* 有环路检测；
* Root Goal Owner 不变化。

否则我不会允许它进入核心业务。

---

### 7. 必须建立上线前评测

每个数字员工上线前，至少需要一套测试集：

* 正常问题；
* 模糊问题；
* 无数据问题；
* 冲突知识问题；
* 越权请求；
* Prompt Injection；
* 高风险操作；
* 知识源失联；
* Tool 调用失败；
* 审批拒绝；
* 配额超限。

并形成版本之间的评测对比。

---

### 8. 数据和模型必须可替换

企业不会愿意被锁死在一个模型或一个知识库中。

我会要求：

* 模型配置可替换；
* Knowledge Adapter 可扩展；
* Artifact 可导出；
* Audit 数据归企业所有；
* 岗位模板和 Skill 可迁移；
* 停止合作后可以完整撤销凭证；
* 不把企业数据用于外部训练。

NIST 的 AI 风险管理框架也强调，AI 风险管理需要围绕组织目标和优先级，贯穿设计、开发、使用和评估，而不只是上线前做一次安全检查。([NIST][5])

# 七、我建议最先试点的三个数字员工

## 试点一：企业经营情报员工

工作：

* 汇总内部经营数据；
* 监测行业和竞争对手；
* 生成管理周报；
* 标记异常；
* 给出需要管理者确认的问题。

特点：

* 只读；
* 输出可人工复核；
* 使用频率高；
* 管理层能直接感受到价值。

---

## 试点二：客服工单协调员工

工作：

* 分类工单；
* 查询知识库；
* 汇总客户历史；
* 生成回复草稿；
* 推荐转派部门；
* 标记重大投诉。

初期不允许：

* 直接退款；
* 直接承诺赔偿；
* 直接关闭重大投诉。

价值指标：

* 首次响应时间；
* 平均处理时间；
* 转派准确率；
* 客服人工查询时间；
* 草稿采用率；
* 客户重复来询率。

---

## 试点三：销售方案协作员工

工作：

* 客户研究；
* 历史交流汇总；
* 需求分析；
* 方案初稿；
* 产品资料查询；
* 合规检查；
* 拜访材料准备。

价值指标：

* 方案准备周期；
* 销售人员准备时间；
* 方案采用率；
* 信息错误率；
* 商机转化率；
* 平均成交周期。

这三个场景分别验证：

```text
管理分析能力
客户服务能力
收入增长能力
```

比一开始创建十几个抽象岗位更容易证明商业价值。

# 八、我的采购决策

假设我是企业总经理或 COO，我会作出这样的决定：

> **批准有限范围采购和业务试点，不批准以“大规模替代员工”为目标的全面部署。**

我会给项目设定三个阶段。

### 第一阶段：辅助

数字员工只查资料、分析和生成草稿，人类完成所有外部行动。

### 第二阶段：协同

数字员工可以创建任务、协调工作和写入低风险内部系统，但关键结果仍需验收。

### 第三阶段：受控执行

只有经过长期验证、错误代价可控、规则明确的场景，才允许数字员工在审批和审计下执行写操作。

达到下一阶段的条件不是“技术已经实现”，而是：

* 业务收益已经证明；
* 错误率可接受；
* 风险机制有效；
* 人员已经适应；
* 责任边界清楚。

## 最终评价

我认为这类平台有真实的企业价值，但它最好的定位不是：

> “一个企业可以雇佣很多 AI 员工。”

而是：

> **“把企业知识工作转化为可分配、可执行、可验收、可审计的数字化工作系统。”**

“数字员工”是便于理解的产品外壳，真正形成采购壁垒的会是：

* 岗位治理；
* 知识权限；
* 工具权限；
* 可靠执行；
* 人机责任划分；
* 业务评测；
* 审计和成本控制；
* 从 Goal 到业务结果的完整闭环。

**我愿意采购的不是一个会自主行动的 AI 组织，而是一个在企业制度内工作、随时可以被监督、限制、验收和停止的数字化劳动力系统。**
