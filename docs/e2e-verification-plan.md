# AI 数字员工平台 — 端到端功能验证方案

> 版本：v1.0 | 日期：2026-07-08 | 状态：待执行

## 验证目标

验证平台从配置到运行的完整链路能正确跑通，覆盖 6 条核心路径、12 项平台能力。

## 验证前置条件

| 条件 | 说明 |
|---|---|
| 平台后端服务 | FastAPI 应用正常启动，PostgreSQL / Redis / MinIO 可用 |
| 模型配置 | 至少一个可用的 OpenAI 兼容 LLM（如 DeepSeek） |
| RAGFlow | 至少一个可用的 RAGFlow 实例（链路 5 需要） |
| 管理员账号 | 拥有全部管理权限的测试账号 |
| 测试数据 | 预备测试用的技能包 ZIP、知识文档等素材 |

## 链路 3：委派链路

**验证目标**：编排者 → 执行者的委派能正确创建和追踪，一层深度限制生效。

### 测试步骤

| # | 操作 | 预期结果 | 验证点 |
|---|---|---|---|
| 3.1 | 创建目标运行，分配给 Lead 级员工 | 目标创建成功 | Lead 有委派权限 |
| 3.2 | Lead 员工提出委派建议（给 Staff 员工） | 平台校验委派策略，创建工作项 | 委派授权 |
| 3.3 | 检查执行图 | 新增节点和边，`depth=1`，关系 `delegated_to` | 执行图结构 |
| 3.4 | Staff 员工完成工作项 | 状态 `completed`，产物归属正确 | 执行者完成 |
| 3.5 | Staff 尝试再次委派 | 被拒绝（MVP 只允许一层委派） | 委派深度限制 |
| 3.6 | 查看目标运行执行图 | 完整展示根目标 → Lead → Staff 委派关系 | 执行图可视化 |

### 成功标准

- 步骤 3.2 委派成功，执行图正确记录
- 步骤 3.5 二次委派被拒绝，返回明确错误信息
- 执行图边和节点关系与实际委派一致

## 链路 4：工具网关与审批链路

**验证目标**：业务工具经过平台网关调用，审批流程正确拦截和放行，幂等性生效。

### 测试步骤

| # | 操作 | 预期结果 | 验证点 |
|---|---|---|---|
| 4.1 | 注册业务工具（只读，无需审批） | 工具创建成功，含幂等性策略 | 工具注册 |
| 4.2 | 注册业务工具（写操作，需审批） | 工具创建成功，`approval_required=true` | 审批配置 |
| 4.3 | 尝试发布缺少幂等性策略的工具 | 发布失败，提示缺少幂等性配置 | 幂等性门禁 |
| 4.4 | 配置幂等性策略后发布 | 发布成功 | 幂等性门禁放行 |
| 4.5 | 员工通过网关调用只读工具 | 直接执行，返回 `status=executed`，记录审计 | 只读工具直通 |
| 4.6 | 员工通过网关调用写工具 | 返回 `status=requires_approval`，暂停执行 | 审批拦截 |
| 4.7 | 审批人在审批中心通过审批 | 工具继续执行，记录审批人、时间戳、审计事件 | 审批放行 |
| 4.8 | 再次调用相同写工具（相同参数） | 幂等键命中，`duplicate=true`，返回已有结果 | 幂等性防重 |
| 4.9 | 员工调用未授权工具 | 网关拒绝，记录审计事件 | 权限控制 |
| 4.10 | 检查审批中心列表 | 包含本次审批的完整记录（待处理→已处理） | 审批中心完整性 |

### 成功标准

- 步骤 4.3 幂等性门禁正确阻断
- 步骤 4.6 ~ 4.7 审批拦截 → 放行流程完整
- 步骤 4.8 幂等性防重正确生效
- 步骤 4.9 未授权调用被拒绝并记录审计

### 成功标准

- 步骤 6.5 组织配额硬阻断正确生效
- 步骤 6.6 目标预算隔离（不影响其他目标）
- 步骤 6.7 ~ 6.9 中断检测 → 上下文保留 → 恢复流程完整
- 步骤 6.10 所有关键事件均有不可变审计记录

### 可用的自动化测试技能

#### 1. Dogfood — Web UI 探索性测试

**用途**：对运行中的 Web 应用进行系统化 QA 测试

**能力**：
- 自动导航、截图、检查 JavaScript 控制台错误
- 测试表单验证、交互流程、边界情况
- 生成带截图证据的结构化 bug 报告

**适用场景**：
- 验证管理后台（技能管理、模板管理、员工管理）的 UI 功能
- 检查审批流程、知识预览等交互页面
- 上线前的全面 UI 质量检查

**使用方式**：
```
触发词："dogfood 测试"、"UI 测试"、"Web 应用测试"
```

#### 2. TDD — 测试驱动开发

**用途**：为新功能先写测试再写代码，强制执行红-绿-重构循环

**适用场景**：
- 新增 API 端点（如委派链路、预算管理）
- 修复 bug 时先写复现测试
- 前端组件开发

**使用方式**：
```
触发词："测试先行"、"TDD"、"红-绿-重构"
```

#### 3. Systematic Debugging — 系统化调试

**用途**：4 阶段根因调试法，定位疑难问题

**流程**：
1. **理解**：收集症状、复现步骤
2. **假设**：列出可能原因
3. **验证**：逐一排除
4. **修复**：实施解决方案并验证

**适用场景**：
- 测试中发现的间歇性失败
- 性能回归问题
- 难以复现的 bug

**使用方式**：
```
触发词："调试"、"诊断"、"定位问题"
```

**8 个专业代理可组合使用**：

| 代理 | 职责 | 适用场景 |
