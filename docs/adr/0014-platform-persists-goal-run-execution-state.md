# Platform persists Goal Run execution state

Platform is the source of truth for Goal Run execution state. Hermes Instances execute employee reasoning and tool calls, but Hermes sessions are not the business state source for Goal Runs, Work Items, Artifacts, Approvals, Budgets, or Trace.

Every meaningful state change is written to Platform storage first, then emitted through an Outbox Event for workers to continue execution. The MVP can use PostgreSQL, an outbox table, Worker processes, and Redis Queue; it does not need Temporal at first. Temporal or another Durable Execution engine can be introduced later if long-running workflows, compensation, and recovery complexity exceed the simple worker model.

This keeps approval waits, budget blocks, artifact rework, retries, cancellation, and crash recovery independent of any single Hermes process or conversation session. Hermes run/session IDs remain useful references for debugging and trace correlation, but they are not authoritative business state.

If a Hermes Instance is interrupted while executing a Work Item, the Platform records the Work Item as `runtime_interrupted` or `failed_retryable`, including the Hermes run/session reference, error reason, any partial Artifact, and Tool-call trace. After the instance recovers, the Goal Run state machine decides whether to retry once, requeue the Work Item, assign it to another qualified employee, or escalate to a human. The MVP does not attempt complex checkpoint resume from the Hermes session.

External write Tool calls must be retried only through Platform-controlled idempotency, approval records, and Audit Events. A recovered Hermes session is not enough evidence that repeating a write is safe. By default, the MVP must not automatically retry external write Tools after runtime interruption unless Tool Gateway can prove safe duplicate handling from the Tool Idempotency Policy and existing audit/idempotency records; otherwise the Work Item must go to human review.

All Platform-managed Tools except Hermes built-in tools must define an idempotency policy before publication, so this retry decision is based on recorded Tool metadata rather than ad hoc code paths.

The default idempotency key is generated by Platform Tool Gateway from `employee_id + goal_run_id + work_item_id + tool_id + request_hash`. External-system-specific idempotency keys are mappings of this Gateway-owned key, not Hermes-generated control values.
