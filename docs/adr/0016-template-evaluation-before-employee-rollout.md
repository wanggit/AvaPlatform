# Template Evaluation before employee rollout

MVP evaluates Job Template Versions before rollout instead of maintaining separate test suites for every Digital Employee. Digital Employees inherit their behavior from a published Job Template Version, so template-level evaluation gives better leverage and avoids duplicated evaluation maintenance.

The evaluation set should cover normal cases, ambiguous requests, missing data, conflicting knowledge, unauthorized access, prompt injection, high-risk Tool calls, missing Knowledge Sources, Tool failures, approval denial, and budget limits. A Template Evaluation Run is one execution record of that plan for one Job Template Version. The MVP stores test cases, inputs, expected results, actual results, manual pass/fail, failure reasons, and key assertions. It does not require automatically invoking Hermes for every case; automated scoring, regression comparisons, and richer quality metrics can come later.

Publishing a Job Template Version requires a passing Template Evaluation result. A failed, running, or missing evaluation blocks publication. Likewise, creating a Digital Employee requires selecting a published Job Template Version whose latest Template Evaluation passed; failed, running, or not-started versions cannot be used to create employees.

Digital Employee creation may only set basic Employee Instance Configuration: name, nickname, avatar, department, direct manager, and notes. Narrowing template-provided Knowledge Sources or Tools, and resource strategy such as prewarming, should be maintained after creation in the employee detail or permission configuration surface. Employee-level configuration cannot expand the published Job Template Version's capability or risk boundary: no broader System Prompt, no extra Skills, no extra Tools, no extra Knowledge Sources, no higher max Goal Risk Level, and no higher default Goal Budget.

Employee state and Hermes runtime state must be modeled separately. Employee Lifecycle State is the Platform control-plane lifecycle, such as provisioning, pending activation, active, disabled, rollout failed, or needs review. Hermes Instance Runtime State is the process and health state, such as not started, starting, healthy, unhealthy, recovering, or stopped. Employee Availability State for the directory and scheduler is derived from lifecycle state, runtime state, workload, budget blocking, and maintenance. A running Hermes process must not make a pending-activation employee schedulable, and an active employee with an unhealthy instance should be unavailable until recovery.

For an active employee whose Hermes Instance becomes unhealthy, Platform keeps the employee lifecycle active but changes availability to unavailable and pauses new scheduling. Instance Manager may auto-restart the instance or rerun health checks within a bounded retry policy, using a recovering runtime state while it works. If recovery succeeds, availability returns to idle or busy based on workload. If recovery exceeds the limit, Platform emits an `abnormal_shutdown` Audit Event, notifies administrators and the direct manager, and leaves disablement as an explicit administrator decision.

Creating a Digital Employee is not the same as activating it. After creation, Platform creates an asynchronous Employee Rollout Job rather than blocking the create modal on Hermes startup or model probing. The job renders the Hermes Profile, writes the Job Template System Prompt into `SOUL.md`, distributes Skills, injects Tool and Knowledge Retrieval configuration, issues an Employee Service Token, starts the Hermes Instance, and runs an instance smoke test. Passing that smoke test moves the employee to pending activation; an administrator must explicitly activate the employee before it can receive Goal Runs.

Creating a Digital Employee must not automatically create a default Goal or initial business task. Rollout validation uses Instance Smoke Test or a clearly marked Smoke/Test Run. These calls may be recorded in Token Ledger and Audit Events with a `test` or `smoke` marker for cost tracking and troubleshooting, but they must not create production Goal Runs, Work Items, KPI entries, or business Artifacts.

The UI should show rollout progress, start/end timestamps, failure class, failure reasons, repair guidance, a log summary, the latest smoke-test result, and actions to rerun configuration or smoke testing. Rollout retries must be idempotent and must not leak duplicate Profiles, occupied ports, or active Employee Service Tokens.

Only transient infrastructure failures should be auto-retried, with a retry limit and backoff: Hermes startup timeout, temporary port conflict, transient model API timeout, or temporary Worker failure. Configuration errors must not auto-retry: missing Job Template configuration, invalid model API key, missing or invalid Skill package, disabled Tool or Knowledge Source reference, or System Prompt rendering failure. After a configuration error, an administrator must repair the configuration and manually rerun configuration or smoke testing.

The employee-level smoke test verifies only the instance wiring: Profile rendering, Hermes startup, service token presence, basic health, and the minimum runtime configuration needed for the employee to call Platform-controlled capabilities. It does not duplicate the Job Template Version's evaluation suite.

Production activation requires one controlled real Hermes API / model probe. The probe must use a test session, low token limit, short timeout, and disabled tool calls. This catches model configuration failures such as Base URL, API Key, model name, context-size, and Profile loading problems that a process-level health check would miss. Development or demo environments may allow manual pass, but the result must be labelled as manual and cannot be treated as a successful real probe.

Pending activation is not a serving state. After the smoke test passes, Platform may stop the Hermes Instance or leave it in cold standby to save resources and avoid accidental scheduling. When an administrator activates the employee, Platform must start or reuse the instance and run a lightweight health check before changing the employee to active. Schedulers and Goal assignment must reject pending-activation employees even if a recently tested Hermes process is still running.
