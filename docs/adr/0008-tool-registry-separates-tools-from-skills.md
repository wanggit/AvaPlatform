# Tool Registry separates runtime tools from skills

Tool is a Platform-managed runtime callable capability, while Skill is a Hermes-native capability package distributed as files. Job Templates choose Tools from a Tool Registry instead of storing hardcoded tool strings.

**Why**: Skills and Tools have different lifecycle and risk boundaries. A Skill teaches a Hermes Agent how to perform work and is distributed into the employee Profile. A Tool grants runtime access to an external capability or Platform API, often with authentication, write permissions, approval requirements, and audit obligations. Mixing them would make authorization, review, and auditing unclear.

**Consequences**: Platform maintains a Tool Registry with Tool metadata, invocation type, authentication type, risk level, read/write classification, approval requirement, audit requirement, idempotency policy, and publication status. Job Templates bind only published/enabled Tools as the Tool whitelist. When a Digital Employee is created or updated, Platform injects the allowed Tool configuration into that employee's Hermes Instance/Profile. Knowledge Retrieval Tool is an internal Tool in the registry, with authorization dynamically constrained by the employee's Job Template Knowledge Source entitlement.

Business Tools are separate from Hermes built-in tools. Hermes built-in tools are generic runtime capabilities supplied by Hermes Agent, while Business Tools are Platform-defined integrations for business systems such as CRM, ticketing, email, approvals, knowledge retrieval, and internal APIs. Business Tools use the Platform Tool Gateway by default: Hermes receives only the Tool schema, Platform endpoint, and Employee Service Token. Third-party credentials remain in Platform and are never written into the Hermes Profile. Direct runtime invocation is not the MVP default and must be explicitly marked later if a narrowly scoped built-in or local tool requires it.

Hermes built-in tools may still be audited and constrained, but through Hermes-native control points rather than Platform Tool Gateway. Platform can use Hermes Profile allowlists to decide which built-in tools are exposed, `pre_tool_call` hooks or middleware to block calls that violate preconfigured policy, `post_tool_call` hooks to emit `tool_call` Audit Events, and result transformation hooks for safe summaries or warnings. Built-in tool constraints in the MVP are preconfigured policy labels and human-readable default constraints, not an administrator-authored rule DSL.

All Platform-managed Tools except Hermes built-in tool registrations must have an idempotency policy before publication. Platform Tool Gateway generates the default idempotency key, at minimum from `employee_id + goal_run_id + work_item_id + tool_id + request_hash`. If an external system mandates its own idempotency-key field, Gateway maps or forwards the Platform-generated key to that field; Hermes does not own idempotency. The policy also defines business object id extraction, duplicate handling, result lookup, non-replayable conditions, and retention TTL. Read Tools may use a simple policy that documents retry and side-effect boundaries. Write Tools must prove idempotency through a Gateway test or mock idempotency test; if a write Tool cannot prove safe duplicate handling, it must remain draft or testing and cannot be bound to Job Templates.

Platform does not create Hermes built-in tool implementations. A `hermes_builtin` Tool Definition is a registration of an already discovered Hermes tool, plus Platform-side metadata such as status, risk, read/write classification, audit flag, and default constraint description. If the named tool does not exist in the Hermes runtime registry, it must not be publishable.

Tool Registry records read/write classification and risk level. Read Tools query systems and normally require audit but not approval. Write Tools change external state and require audit by default; high-risk write Tools require Approval. High-risk read Tools, such as access to unmasked customer data, raw audit evidence, or financial data, receive higher audit severity and may require Approval. Job Template Tool binding creates a Tool Entitlement that may narrow parameters beyond the registry default, such as department scope, refund amount limit, draft-only email creation, or allowed customer segment.

Approval is owned by Platform business policy, not by the requesting Hermes Agent. Default approvers are the Digital Employee's direct manager or department owner; high-risk Business Tools may additionally require the Tool Owner; high-risk Hermes built-in tools such as terminal or process management require platform administrator or operations owner approval. The MVP approval surface is a Platform internal Approval Center with approve/deny decisions, comments, expiration, requester context, Tool context, risk reason, parameter summary, and audit references. External IM cards may be added later as notification and action channels.

The approval execution pattern must fit Hermes' current tool loop. Business Tools use the Platform Tool Gateway: when approval is required, the gateway creates an Approval Request and returns a safe `approval_required` Tool result with `approval_id` and `audit_id` instead of calling the business system. Hermes receives that Tool result in conversation context; after approval, the Digital Employee must retry the Tool call with the `approval_id`. Hermes built-in tools use Hermes-native control points: Platform's Hermes plugin can block in `pre_tool_call` and emit a safe Tool result, and Hermes' dispatcher appends blocked Tool results back to the model context; Hermes also already has a gateway approval queue for dangerous command approvals that can be reused or mirrored if a blocking in-flight approval flow is required. The MVP should prefer the explicit retry-after-approval pattern for clarity and auditability.

Business Tool integration types are constrained in the MVP. `platform_api` calls Platform-owned APIs such as knowledge retrieval, approval creation, and employee directory search. `http_api` calls controlled enterprise HTTP APIs with configured base URL, path, method, headers, credential reference, request schema, and response schema. `mcp_server` is reserved for connecting a controlled MCP server and may remain a configuration placeholder in the MVP. The MVP does not support administrator-uploaded Python/JavaScript tool code, arbitrary script transforms in the UI, or Hermes directly executing business-system SDKs.

The Tool edit form must vary by integration type. `hermes_builtin` is a registration flow for a discovered Hermes tool: it shows Hermes name, toolset, read/write class, description, and schema as read-only runtime facts, then lets Platform configure only governance metadata such as enabled/disabled/deprecated status, risk level, audit requirement, and preconfigured constraint policy labels. Endpoint fields, Credentials, editable schema, free-text rule DSL, and ordinary publish/draft lifecycle controls are not applicable. `platform_api` selects or defines a Platform-owned API endpoint and may use Platform internal credentials only. `http_api` requires Base URL, path, method, Credential reference, and request/response schema. `mcp_server` is a lightweight registry entry in the MVP, recording server name, tool name, risk, audit, and constraint metadata while detailed MCP server configuration is handled separately.

Business-system credentials are managed inside this Platform, not through a third-party secret manager in the MVP. Platform stores Credential records for API keys, OAuth tokens, Basic Auth values, webhook secrets, and similar business-system credentials. Tool Definitions reference Credentials by `credential_ref`; they do not store plaintext secrets. Job Templates cannot change Credentials, and Hermes Profiles never receive business-system credentials. Rotating a Credential updates future Platform Tool Gateway calls without changing Job Templates. Audit Events may record `credential_ref` and masked metadata, but never the secret value.

Tools have their own lifecycle: `draft`, `testing`, `published`, `disabled`, and `deprecated`. Draft Tools are editable and cannot be bound to Job Templates. Testing Tools support connection tests, schema tests, mock calls, and idempotency tests. Published Tools can be bound to Job Templates. Disabled Tools cannot be called at runtime and should raise publish-time warnings if still referenced. Deprecated Tools may keep running for migration but should not be newly bound. Publishing requires an active Credential when needed, complete request/response schema, Approval and Audit settings for write or high-risk Tools, a successful Gateway test or mock test, complete idempotency policy for non-`hermes_builtin` Tools, and complete HTTP configuration for `http_api` Tools.

Tool whitelist changes follow the Job Template Version publication model. Draft Job Template changes do not affect runtime. Publishing a new Job Template Version computes affected Digital Employees and shows Tool add/remove diffs, risk-level changes, and whether write or high-risk Tools are involved. Active employees receive a Tool sync that updates Profile tool schema, Platform endpoint, and allowed Tool list, then attempts Hermes tool reload; if reload is unavailable or fails, Platform restarts the Hermes Instance. Credential rotation does not require a Job Template publish or Hermes reload because Platform Tool Gateway uses the current Credential. If a Tool is disabled, Platform Tool Gateway immediately rejects runtime calls, and templates or employees still referencing it show risk warnings.

Every Business Tool call emits a `tool_call` Audit Event. The event records employee, tool, tool version, Job Template version, request id, timestamps, status, latency, read/write class, risk level, and approval id when present. Low-risk read Tools record metadata and masked summaries rather than full request and response bodies. Write Tools record request summary, business object ids, change result, and external system response ids. High-risk Tools record `evidence_refs` for review, with masked UI display by default. Failed calls are audited, especially authorization failures, missing approval, out-of-entitlement parameters, and external system errors. Normal successful Tool calls do not affect KPI by default; red-line, unauthorized, rejected, or repeated failure scenarios may affect KPI through Audit Rules.

Digital Employee Tool permissions inherit from the published Job Template. Employee-level configuration cannot add Tools outside the template's Tool Entitlements. It may only disable a template-provided Tool for that employee or narrow the Tool constraints further. Temporary high-risk Tool grants must go through Approval, emit Audit Events, and carry an expiration time; the MVP can defer temporary grants.

Platform Tool Gateway returns standardized, safe errors to Hermes. It does not expose raw third-party errors, credentials, or internal stack traces. Standard error codes include `tool_not_allowed`, `tool_disabled`, `credential_unavailable`, `approval_required`, `approval_denied`, `constraint_violation`, `external_system_error`, `tool_timeout`, `rate_limited`, and `audit_required_failed`. Responses include a safe message, `retryable`, optional `approval_id`, `audit_id`, and safe details. For high-risk Tools, if required audit writing fails, the call must not proceed.
