# Audit rules are observability, not enforcement

Audit Rules define how Audit Events are classified, notified, reviewed, and counted into KPI. They do not replace the execution logic of Red Line, Approval, Organization Quota, Goal Budget, or other runtime mechanisms.

**Why**: Red Line, Approval, Organization Quota, Goal Budget, and runtime control each have their own domain-specific state and failure modes. If Audit Rules become a second enforcement engine, the platform will have duplicated decision paths, unclear ownership, and hard-to-debug conflicts.

**Consequences**: Red Line triggers, approvals, escalations, budget blocks, abnormal shutdowns, and sensitive operations must emit immutable Audit Events. Audit Rules can set severity, notify receivers, require human review, mark events as KPI-affecting, and provide suggested follow-up actions. They cannot directly stop employees, change permissions, publish templates, approve actions, or override budget behavior.

Audit Events are append-only. Their core fields, original payload, matched `rule_id`, and matched `rule_version` cannot be edited after creation. Administrators can only append Audit Dispositions such as false positive, confirmed, handled, no action needed, or escalated. Later Audit Rule changes do not rewrite historical events; they affect only newly created events. Sensitive evidence may be masked in the UI, but the original evidence remains retained behind appropriate permissions.

Audit uses a unified event model. Different audit scenarios are represented by `event_type`, optional subtype, and structured `payload`, instead of separate tables and screens for each category. The MVP event types include `red_line_triggered`, `approval_requested`, `approval_decided`, `escalation_created`, `abnormal_shutdown`, `sensitive_operation`, `budget_blocked`, `knowledge_preview`, `template_published`, and `skill_package_changed`. This keeps listing, filtering, detail view, review, retention, and export behavior consistent across event categories.

Audit Rule matching is intentionally simple in the MVP. Rules are configured from explicit conditions such as `event_type`, `severity`, `department`, `employee_id`, `grade`, `resource_type`, `actor_type`, `contains_sensitive_data`, `red_line_category`, `operation_name`, business-hours flags, and `estimated_usage`. Rule actions are limited to setting severity, notification behavior, receivers, review requirement, KPI impact, retention period, and suggested follow-up actions. The MVP does not allow administrator-defined JavaScript, Python, expression scripts, arbitrary external calls, or direct runtime enforcement actions from Audit Rules. More expressive rule engines can be added later if real demand appears.

Audit notifications start with in-platform notification records. Audit Rules may target Admins, the employee's direct manager, department owners, or specified users/roles. The UI can expose email and Feishu/DingTalk webhook channels as configuration placeholders, but the MVP does not need to deliver external IM notifications. Audit notification failure does not block the business flow and does not change the original Audit Event; it is recorded as notification status or a separate `notification_failed` Audit Event.

KPI impact has conservative defaults. `red_line_triggered`, `escalation_created`, Goal-level `budget_blocked`, and employee-attributable `abnormal_shutdown` events affect KPI by default. Organization-level `budget_blocked` is retained for cost control audit but does not penalize an individual employee by default. `approval_decided` affects KPI only when the decision is rejected because the employee attempted an out-of-policy action or lacked sufficient evidence. `approval_requested`, approved `approval_decided`, `knowledge_preview`, `template_published`, `skill_package_changed`, and ordinary sensitive admin operations are retained for audit but do not affect KPI by default. Audit Rules may override `kpi_affecting` for specific scenarios.

`abnormal_shutdown` is reserved for unexpected runtime failure: Hermes Instance process exit, repeated health-check failure, start/restart failure, OOM kill, or external process-manager termination. Admin stop is a `sensitive_operation` with subtype `employee_stopped_by_admin`. Budget blocking is `budget_blocked`. Planned reloads caused by template publication or configuration change are `template_published` or `sensitive_operation`, not abnormal shutdown. Whether an abnormal shutdown affects KPI depends on attribution: infrastructure-caused failures do not penalize the employee, while repeated failures caused by that employee's profile, Skill, or tool configuration may affect KPI.

Sensitive operations are split into admin-sensitive and employee-sensitive operations. Admin-sensitive operations include viewing unmasked evidence, exporting audit logs, publishing templates, uploading or publishing Skill Packages, changing Knowledge Connections or Model Configurations, adjusting Organization Quota or Goal Budget, starting/stopping employees, clearing `budget_blocked`, and rotating Employee Service Tokens. Employee-sensitive operations include attempted external write actions such as sending email, updating CRM, submitting tickets, or calling write-capable tools. The MVP audit page focuses on admin-sensitive operations; employee-sensitive operations can be represented through Approval and Tool-related Audit Events first.

Audit retention is simple in the MVP. The default retention period is 180 days. High-severity events such as red line triggers, high-risk sensitive operations, employee-attributable abnormal shutdowns, and budget blocking are retained for 365 days. Audit Rules may choose retention only from preset values: 90 days, 180 days, 365 days, or permanent. Audit export is itself a sensitive operation and must emit an Audit Event. Exports are masked by default; exporting unmasked evidence requires Admin permission, second confirmation, and a stated reason. The MVP does not require external archive storage or WORM storage, but the design should leave room for those later.

The Audit Rule page should support lightweight rule preview. While editing a rule, an administrator can choose a recent or mock Audit Event and test whether the rule matches. The preview shows matched conditions, resulting severity, notification behavior, receivers, review requirement, KPI impact, and retention period. Preview does not write a new Audit Event, send notifications, or change KPI.

The MVP keeps audit in one management page. The existing "Audit Rules" menu hosts tabs for Audit Events, Audit Rules, Notification Records, and Retention Policy. Events and rules are strongly related, and keeping them together reduces navigation cost while the product is still small. If event volume or operational ownership grows later, the page can split into an Audit Center and an Audit Rules settings page.
