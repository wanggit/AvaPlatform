## 背景

本平台是一个单组织人工智能数字员工管理系统，底层运行时使用 Hermes Agent。Hermes 负责每一个数字员工的执行环境；平台负责组织侧控制面，包括岗位模板、数字员工、员工目录、目标委派、工具治理、知识库集成、预算、审计、关键指标和管理后台。

当前信息分散在 `CONTEXT.md`、`start.md`、`docs/adr/` 和 `web/` 原型中。本设计把这些结论合并成一份可执行的 OpenSpec 变更。`hermes-agent-2026.6.19/` 是只读上游源码，平台不修改 Hermes 源码，只通过 Hermes 配置档案、配置档案文件、可用的钩子或中间件、以及 Hermes 接口服务完成集成。

关键约束如下：

- 产品不是多租户系统。
- 只有一个管理后台。
- 每一个数字员工一一对应一个 Hermes 配置档案和一个 Hermes 实例。
- 平台是业务状态的事实来源；Hermes 的运行记录和会话记录只作为运行时遥测与追踪上下文。
- 岗位模板版本是数字员工能力、风险、工具、知识和人设的边界。
- 业务工具和密钥由平台管理；Hermes 内置工具仍由 Hermes 执行，但通过白名单、钩子、中间件、默认约束和审计元数据治理。
- RAGFlow 是第一个知识库适配器；平台不自研知识库，也不把 RAGFlow 凭据下发给 Hermes。

## 目标与非目标

目标：

- 正式固化单组织人工智能数字员工平台的最小可行版本需求合同。
- 保留 `CONTEXT.md` 中已经形成的领域词汇。
- 定义数字员工的生命周期、运行时状态、可调度状态、部署、冒烟测试和激活语义。
- 定义由平台持久化管理的目标运行、工作项、执行图、审批、预算、产物和运行中断恢复。
- 定义工具注册、平台工具网关、业务工具幂等性、审批和审计边界。
- 定义知识源授权和平台代理检索。
- 定义组织配额、目标预算、令牌台账、关键指标和业务结果指标。
- 定义最小可行版本管理后台原型必须覆盖的页面和状态。

已接受的调整：

- 产品形态固定为单组织、单管理后台，不实现多租户、超级管理后台或租户后台。
- 每个数字员工对应一个 Hermes Profile 和一个 Hermes Instance；Platform 不修改 Hermes 上游源码。
- Platform 后端采用 FastAPI，第一阶段采用模块化单体，而不是微服务。
- Platform 使用 PostgreSQL 作为事实库，Redis 作为队列、缓存和锁，MinIO/S3 兼容对象存储保存技能包、产物附件和审计证据。
- 后台异步工作由 Platform Worker 执行，覆盖员工上岗、Hermes 调用、知识同步、审批恢复、预算和审计处理。
- RAGFlow 是首个知识库适配器；RAGFlow 凭据只保存在 Platform，不下发到 Hermes Profile。
- Embedding 开发配置可使用本机 Ollama 向量模型；LLM 开发配置使用 OpenAI 兼容模型配置；Rerank 可在第一阶段关闭。
- SMTP 用于审批、预算预警和审计通知；真实 SMTP、RAGFlow 和模型密钥只进入本地环境变量或部署环境，不写入仓库。

非目标：

- 不做多租户、租户管理后台、超级管理后台或租户级模板复制。
- 不在平台内自研知识库。
- 最小可行版本不实现全自动 Hermes 模板评估引擎。
- 不做完整商业智能系统。
- 第一阶段后端不引入 Temporal 等复杂持久工作流引擎。
- 不允许管理员上传任意工具代码，不支持自由脚本转换，不让 Hermes 直接调用业务系统 SDK。
- 不实现复杂的 Hermes 会话级断点续跑。

## 设计决策

### 单后台与单组织

平台应建模为一个组织下的一套管理后台。岗位模板、技能、工具、知识源、模型配置、预算和审计配置都直接归属于这个组织。

曾考虑过多租户 SaaS、租户后台和超级管理后台。该方案已被放弃，因为它会引入模板复制、租户隔离、租户配额和跨租户审计复杂度，不符合当前产品目标。

### 技术架构采用模块化单体

第一版 Platform 后端采用 FastAPI 模块化单体，并拆出独立 Worker 进程处理异步任务。前端继续使用 React、TypeScript 和 Ant Design。PostgreSQL 是业务事实库；Redis 承载队列、缓存和分布式锁；MinIO 或 S3 兼容对象存储保存上传技能包、业务产物附件、导入文件和审计证据。

模块边界按业务能力组织，包括岗位模板、数字员工、目标运行、审批、工具、知识、模型、预算、审计和指标。模块内部可以保持清晰边界，但第一阶段不拆成微服务。后续只有在某个能力出现独立扩缩容、独立团队交付或清晰稳定的跨服务契约时，才评估服务拆分。

推荐运行形态如下：

```text
React 管理后台
        |
FastAPI Platform API
        |
PostgreSQL + Redis + MinIO/S3
        |
Platform Worker
        |---------------- Hermes Runtime Manager
        |---------------- Platform Tool Gateway
        |---------------- RAGFlow Adapter
        |---------------- Budget / Audit / Evaluation Jobs
        |
Hermes Profile + Hermes Instance per Digital Employee
```

开发环境应使用平台专用 PostgreSQL、Redis 和 MinIO 容器，避免复用其他项目或 RAGFlow 自带依赖。RAGFlow 可以作为外部已部署服务接入。Hermes 测试必须创建独立 Profile，不能修改用户默认 Hermes Profile。

第一阶段开发配置基线：

- Platform API 使用非默认端口，例如 `8010`，避免与现有本机服务冲突。
- PostgreSQL、Redis 和 MinIO 使用平台专用端口与数据卷。
- RAGFlow 使用外部 Base URL 和 API Key 环境变量。
- Ollama Embedding 使用 OpenAI 或 Ollama 兼容适配器统一封装到模型配置。
- DeepSeek 或其他 OpenAI 兼容 LLM 作为模型配置记录，不把 API Key 写入仓库。
- SMTP 配置只以环境变量形式注入，用于审批和告警通知。

### 岗位模板版本是能力边界

数字员工只能从已发布且模板评估通过的岗位模板版本创建。创建员工时只能填写员工实例配置。员工创建后可以收窄继承来的工具或知识权限，但不能扩大能力、知识、工具、风险等级、预算或系统提示词 职责范围。

曾考虑允许特权用户从未验证模板创建员工。该方案已被放弃，因为它会削弱安全门禁，也会让员工行为难以解释。

### 模板评估是岗位模板门禁，不是员工测试

模板评估作用于岗位模板版本。最小可行版本阶段记录测试用例、输入、期望结果、实际结果、关键断言、人工通过或失败结果、失败原因和评估人。后续可以增加自动运行 Hermes 测试用例。员工级验证只保留实例冒烟测试和人工冒烟或测试运行。

曾考虑为每个数字员工维护独立测试套件。该方案已被放弃，因为员工行为继承自岗位模板版本，员工级测试套件会造成重复维护。

### 数字员工部署是异步流程

创建数字员工时，系统先创建员工记录和员工部署任务，然后立即返回。部署流程负责渲染配置档案、签发员工服务令牌、下发技能、注入工具和知识配置、启动 Hermes，并运行实例冒烟测试。部署成功后员工进入待激活状态，必须由管理员显式激活。

曾考虑让创建弹窗等待 Hermes 启动和模型探测完成。该方案已被放弃，因为启动和探测耗时长、失败点多，不适合同步阻塞。

### 状态模型拆分

数字员工状态拆为三类：

- 员工生命周期状态：业务控制面状态。
- Hermes 实例运行时状态：进程和健康状态。
- 员工可调度状态：目录和调度器使用的派生状态。

这种拆分可以避免“进程运行中”被误认为“员工已可接生产任务”，也避免运行时短暂异常直接改变员工业务生命周期。

### Hermes 集成保持配置档案与接口方式

平台应把系统提示词 渲染到配置档案的 `SOUL.md`，把技能包下发到员工配置档案的 `skills/`，把平台工具配置和员工服务令牌引用写入配置档案运行配置，并调用 Hermes 接口服务做冒烟测试和工作执行。

曾考虑派生修改 或修改 Hermes 核心。该方案已被放弃，因为当前约束明确要求不修改 Hermes 上游源码。

### 平台拥有执行状态

目标运行、工作项、产物、审批、预算和追踪状态必须先持久化到平台存储，再通过出站事件或工作进程推动下一步动作。Hermes 的运行或会话标识只作为追踪引用，不作为工作流事实来源。

曾考虑把 Hermes 会话当作工作流状态。该方案已被放弃，因为崩溃恢复、审批、预算和外部写操作幂等性都需要平台拥有持久状态。

### 目标驱动委派与受限执行图

人或业务系统创建目标。根目标负责人始终对业务结果负责。数字员工可以提出委派建议，但由平台创建授权工作项、执行委派策略、维护执行图、控制重试、取消和恢复。最小可行版本只允许一层编排者到执行者的委派关系，默认不允许被委派员工继续再委派。

目标执行过程中的人工审批应进入统一的“审批中心”页面。审批中心按待处理、已处理、已过期、已拒绝分组展示所有需要人类决策的事项，包括工具调用审批、预算超限处理、异常停机处置、产物验收和高风险操作复核。目标运行详情页、工作项详情页、工具调用详情页只提供上下文入口和局部快捷操作，最终审批记录仍归档到审批中心。

### 工具治理通过平台工具网关

业务工具必须经过平台工具网关。Hermes 只拿到工具结构定义、平台接口端点和员工服务令牌，不能拿到外部业务系统凭据。除 Hermes 内置工具登记外，所有平台管理的业务工具在发布前都必须声明幂等性策略。默认幂等键由网关根据员工、目标运行、工作项、工具和请求摘要生成。

Hermes 内置工具单独治理：通过已发现工具登记、配置档案白名单、默认约束、钩子或中间件、审计元数据来约束。它们不走平台工具网关的幂等性发布门禁。

### 知识检索通过平台适配器

数字员工通过平台知识适配器和知识检索工具访问知识。RAGFlow 凭据只保存在平台。岗位模板版本授予知识源访问范围；员工级配置只能继续收窄。知识检索结果应返回引用信息和审计标识。

### 预算模型只保留组织配额和目标预算

最小可行版本的硬控制只有组织配额和目标预算。部门、数字员工和工作项只作为用量分析维度，不作为独立阻断预算。令牌台账记录模型供应商或 Hermes 返回的用量证据；平台不把 Hermes 内部累计计数当作配额台账。

### 审计规则用于观察和处置，不直接执法

审计事件是不可变事实。审计规则负责分类、通知、要求复核、设置保留策略和影响指标。审计规则本身不直接停用员工、不变更权限、不发布模板、不批准操作，也不覆盖预算行为。

## 风险与取舍

- RAGFlow 的文档解析能力较强，但资源占用高于轻量组件。最小可行版本先支持一个活动知识连接，同时保留平台侧供应方抽象。
- 人工模板评估可能不一致。必须要求记录明确用例、期望结果、实际结果、关键断言、通过或失败结论和失败原因；后续再增加自动回归。
- Hermes 冒烟测试会消耗令牌。冒烟测试应使用测试会话、低令牌上限、短超时、禁用或限制工具调用，并在令牌台账中标记用途。
- 运行中断可能造成外部写操作重复。除非工具网关能基于幂等策略、审批和审计记录证明安全，否则默认不自动重试写操作。
- 这份能力规格覆盖面较宽。先保持需求完整和可追踪，后续如果实现团队需要独立生命周期，可以再拆分能力。
- 第一阶段不引入 Temporal。先使用 PostgreSQL、出站事件、工作进程和 Redis Queue；如果恢复复杂度上升，再评估持久工作流引擎。
- Hermes 内置工具治理弱于业务工具网关治理。应保持内置工具与业务工具分离，默认不暴露高风险内置工具。

## 迁移计划

1. 将此 OpenSpec 变更作为最小可行版本基线需求合同。
2. 在归档前，继续把 `CONTEXT.md`、`start.md` 和 ADR 作为辅助领域与设计上下文。
3. 根据规格补齐原型，尤其是工具幂等性、模板评估记录、员工部署、知识源和预算页面。
4. 后端实现按 `tasks.md` 的顺序切分，在模拟兼容接口背后逐步替换原型数据。
5. 每个实现切片都用 OpenSpec 场景和现有原型行为校验。
6. 实现完成并评审通过后，归档此 OpenSpec 变更，并同步到主规格。

回滚策略：因为本次主要是规划和规格变更，归档前的回滚就是删除或废弃此 OpenSpec 变更。后续实现阶段，每个后端切片都应保持现有原型数据可被替换，而不是一次性破坏原型。

## 待确认问题

- 第一版生产后端最终采用 `start.md` 中建议的 FastAPI，还是改用其他框架？
  - 已确认：平台后端继续使用 FastAPI。
  - 说明：这是 Platform 后端选型；Hermes 自身的 API Server 源码使用 `aiohttp` 实现，Platform 不需要也不应该因此改变自身后端框架。

- 哪个 Hermes 接口端点 作为每次工作执行和用量提取的标准调用路径？
  - 结论：生产工作执行的标准调用路径采用 Hermes API Server 的 `POST /v1/runs`。
  - 配套接口：`GET /v1/runs/{run_id}` 查询状态，`GET /v1/runs/{run_id}/events` 订阅结构化事件，`POST /v1/runs/{run_id}/approval` 响应 Hermes 侧待审批请求，`POST /v1/runs/{run_id}/stop` 中断运行。
  - 依据：`hermes-agent-2026.6.19/gateway/platforms/api_server.py` 明确暴露上述端点，并在 run 完成事件和状态中返回 `input_tokens`、`output_tokens`、`total_tokens` 用量。`/api/sessions/{session_id}/chat` 更适合交互式会话，不适合作为平台生产目标运行的主通道。

- 平台渲染工具结构定义、员工服务令牌引用和知识检索工具配置时，配置档案文件格式的最终结构定义是什么？
  - 建议：不要把平台私有业务字段直接塞进 Hermes 原生 `config.yaml` 的未定义区域。采用“原生 Hermes 文件 + 平台自有清单文件”的组合：
    - `SOUL.md`：只放岗位模板版本渲染出的系统提示词和岗位人设。
    - `config.yaml`：只写 Hermes 原生可识别配置，例如模型、供应方、`platform_toolsets.api_server`、`mcp_servers`、技能启停配置。
    - `.env`：只写 Hermes 运行所需的本地环境变量；外部业务系统凭据不写入此文件，仍保存在 Platform。
    - `skills/`：放岗位模板版本绑定的技能包。
    - `platform/employee.yaml`：平台自有清单，记录 `employee_id`、`job_template_version_id`、员工服务令牌引用、工具白名单摘要、知识源授权摘要、预算上下文、审计上下文和回调地址。
  - 工具与知识检索接入建议：Platform 暴露一个或多个 MCP server 或 Hermes 可识别的工具集入口，Hermes Profile 只配置这些入口；实际工具 schema、权限校验、审批、幂等、RAGFlow 检索和凭据使用都在 Platform 侧完成。

- 首批生产种子数据中要内置哪三个试点岗位模板？每个模板必须包含哪些模板评估用例？
  - 已确认方向：MVP 需要至少三个 Pilot Job Template，用于验证业务价值。具体岗位名称、评估用例和业务结果指标可在种子数据任务中确定，不阻塞技术架构和开发环境准备。
- 第一批可写业务工具会对接哪些外部业务系统？
  - 明确推迟：当前先实现 Tool Registry、Credential、幂等策略、审批和网关抽象；真实业务系统连接可在首个试点岗位确认后再接入。
