## 为什么要做

人工智能数字员工平台的产品方向已经经过多轮讨论，从早期多租户设想调整为单组织、单后台的管理系统。当前结论分散在 `start.md`、`CONTEXT.md`、`docs/`、原型页面和会话记录中，还没有形成一份可被 OpenSpec 插件识别、可驱动后续实现的正式需求规格。

本次变更用于把这些业务决策整理成一个 OpenSpec 变更：明确平台边界、Hermes 集成方式、岗位模板、数字员工生命周期、目标运行、工具、知识库、预算、审计、指标和原型页面要求。

## 变更内容

- 建立一个正式的 OpenSpec 能力：人工智能数字员工平台。
- 明确系统是单组织、单管理后台，不再做多租户、超级管理后台或租户后台拆分。
- 明确每一个数字员工对应一个 Hermes 实例和一个 Hermes 配置档案。
- 明确岗位模板版本、模板评估、数字员工创建门禁和发布机制。
- 明确数字员工创建后的异步部署、实例冒烟测试、人工激活、生命周期状态、运行时状态和可调度状态。
- 明确系统提示词写入 Hermes 配置档案、技能包下发、员工服务令牌、Hermes 接口服务调用和运行时恢复边界。
- 明确目标运行、工作项、执行图、委派、运行中断、产物验收和重试边界由平台持久化管理。
- 明确技能管理、工具注册、平台工具网关、业务工具幂等性、Hermes 内置工具治理、审批和审计要求。
- 明确使用 RAGFlow 作为首个外部知识库，对接方式为平台代理检索，不自研知识库。
- 明确模型配置、组织配额、目标预算、令牌台账、用量分析、关键指标、业务结果指标和审计规则。
- 明确原型页面需要体现的管理入口、字段和状态。
- **破坏性变更**：废弃此前所有多租户、租户模板复制、租户配额、超级管理后台、租户管理后台相关要求。
- **破坏性变更**：创建数字员工不得直接创建生产目标运行，也不得自动让员工上岗。

## 能力

### 新增能力

- `ai-digital-employee-platform`：单组织人工智能数字员工平台的端到端需求，包括配置管理、数字员工生命周期、Hermes 运行时集成、目标协作、工具治理、知识检索、预算、审计、指标和管理后台原型行为。

### 修改能力

- 无。

- 已准备三套试点岗位模板种子数据：客服工单协调员工、销售方案协作员工、招聘初筛协作员工；每套都包含模板评估用例和业务结果指标。

- 已验证组织配额、目标预算、令牌台账聚合、业务结果指标采集和审计规则评估。
- 当前验证命令：`uv run pytest`、`npm run lint`、`npm run build`、`openspec validate formalize-digital-employee-platform-requirements --strict`。
